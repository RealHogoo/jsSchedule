# 노드 기능 문서

## 목적

노드는 태스크 내부의 세부 구조를 트리 형태로 관리하기 위한 단위다.
사용자는 프로젝트와 태스크를 선택한 뒤, 해당 태스크 아래에서 노드를 생성, 수정, 이동, 삭제할 수 있어야 한다.

노드는 단순 계층 구조만 가지는 것이 아니라, 태스크 타입별로 정의된 통계 항목 값을 함께 입력할 수 있어야 한다.
이 값은 이후 집계 화면과 통계 산출의 원천 데이터로 사용한다.

## 화면 방향

- 별도 노드 전용 메인 화면은 두지 않는다.
- `task.html` 내부 인라인 작업영역에서 노드 트리와 노드 상세 편집을 함께 제공한다.
- 기본 흐름은 `프로젝트 선택 -> 태스크 선택 -> 노드 트리 편집`이다.
- 태스크를 벗어나 독립적으로 노드를 관리하는 UX는 1차 범위에서 제외한다.

## 주요 화면

- 태스크 목록 하단 노드 작업영역
- 노드 트리 패널
- 선택 노드 상세/등록 패널
- 태스크 타입별 통계 항목 설정 화면

## 노드 작업영역

- 상단에는 현재 프로젝트, 현재 태스크, 현재 선택 노드를 표시한다.
- 좌측은 노드 트리, 우측은 선택 노드 편집 폼으로 구성한다.
- 데스크톱은 2컬럼, 모바일은 상하 스택 구조로 전환한다.
- 신규 등록과 수정은 동일한 우측 패널에서 처리한다.

## 노드 트리

- 각 노드는 `펼치기`, `노드명`, `유형`, `상태`, `액션` 정도만 표시한다.
- 액션은 `하위 추가`, `형제 추가`, `수정`, `삭제`를 기본으로 제공한다.
- 선택 노드는 시각적으로 명확히 강조한다.
- 트리 펼침 상태는 저장 후에도 유지한다.

## 이동 정책

### 1차 범위

- 드래그 앤 드롭 이동은 지원한다.
- 같은 태스크 내부에서만 이동 가능하다.
- 부모 변경 이동과 형제 순서 변경을 모두 허용한다.
- 자기 자신 또는 자기 하위 노드 아래로는 이동할 수 없다.

### 예외 및 제약

- 루트 노드는 다른 태스크로 이동할 수 없다.
- 삭제 시 자식 노드가 존재하면 기본은 삭제 제한으로 처리한다.
- 필요 시 관리자 권한에서만 하위 포함 삭제를 별도 제공할 수 있다.

### 실패 처리

- 드래그 후 즉시 화면 반영하되, 저장 실패 시 원래 위치로 롤백한다.
- 서버는 이동 요청 시 부모 변경과 정렬순서 재계산을 함께 처리한다.

## 노드 상세 폼

- 노드명
- 부모 노드
- 정렬순서
- 노드 유형
- 설명
- 사용 여부
- 생성일/수정일 메타 정보

## 통계 항목 관리 방향

노드 테이블에 고정 컬럼 10세트를 직접 두는 방식은 1차안으로는 보류한다.
대신 태스크 타입별 항목 정의와 노드별 값 저장을 분리한다.

이유는 다음과 같다.

- 태스크 타입마다 쓰는 항목이 다르다.
- 항목 수 제한이 바뀌어도 스키마 변경 영향을 줄일 수 있다.
- 통계 대상 항목만 선별하기 쉽다.
- 화면에서 `최대 10개` 제한을 두더라도 내부 구조는 유연하게 유지할 수 있다.

## 통계 항목 규칙

- 태스크 타입별로 최대 10개의 항목을 정의할 수 있다.
- 각 항목은 다음 속성을 가진다.
  - 항목명
  - 통계 포함 여부
  - 값 슬롯 3개
  - 정렬순서
  - 사용 여부
- 값 슬롯 3개는 공통 구조를 사용하되, 라벨은 타입별 정책에 따라 확장 가능하도록 둔다.

## 화면 UX 원칙

- 사용자는 현재 어떤 노드를 기준으로 추가하는지 항상 알아야 한다.
- `하위 추가`인지 `형제 추가`인지 저장 전에 명확히 보인다.
- 저장 후 트리와 상세 폼은 즉시 동기화한다.
- 노드 이동은 마우스 드래그로 제공하되, 보조 수단으로 `위로`, `아래로`, `들여쓰기`, `내어쓰기`를 둘 수 있다.
- 모바일에서는 드래그보다 액션 버튼 우선 전략을 고려한다.

## API 초안

- `POST /node/tree.json`
  - 태스크 기준 트리 조회
- `POST /node/detail.json`
  - 단일 노드 상세 조회
- `POST /node/save.json`
  - 신규 등록 및 수정
- `POST /node/move.json`
  - 부모 변경 및 정렬 이동
- `POST /node/delete.json`
  - 단일 노드 삭제
- `POST /task-type/metric/list.json`
  - 태스크 타입별 항목 정의 조회
- `POST /task-type/metric/save.json`
  - 항목 정의 저장
- 상세 요청/응답 규격은 [node-api.md](D:/MSA_project/ADMIN_project/schedule-service/docs/node/node-api.md)에 정리한다.

## 구현 단계 제안

### 1단계

- 노드 트리 조회
- 노드 등록/수정/삭제
- 태스크 타입별 통계 항목 정의
- 노드 상세에서 항목 값 3개 입력

### 2단계

- 드래그 앤 드롭으로 부모 변경/순서 변경
- 저장 실패 롤백
- 트리 펼침 상태 유지

### 3단계

- 통계 집계 화면
- 태스크 타입별 항목 라벨 확장
- 권한별 삭제 정책 분기

## SQL 초안

- PostgreSQL 기준 DDL 초안은 [node-schema.sql](D:/MSA_project/ADMIN_project/schedule-service/docs/sql/postgres/node-schema.sql)에 정리한다.
- 현재 파일은 문서 초안이며, 실제 적용 전 기존 데이터 이행 방식과 제약조건 충돌 여부를 검토해야 한다.

## 고려 사항

- 태스크 타입 변경 시 기존 노드의 통계 항목 값 유지/초기화 정책이 필요하다.
- 통계 제외 항목은 입력은 허용하되 집계에서는 제외한다.
- 항목 10개 제한은 UX 제약이며, DB 구조는 유연하게 유지한다.
- 초기 버전에서는 드래그 앤 드롭과 버튼 이동을 함께 제공하지 말고 우선순위를 정해야 한다.
